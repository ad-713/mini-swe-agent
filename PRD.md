# Product Requirements Document: Minimal GitLab SWE Agent CLI

## 1. Executive Summary
The Minimal GitLab SWE Agent is a highly performant, local CLI-based AI coding assistant derived from the mini-swe-agent architecture. It is purpose-built to autonomously resolve assigned GitLab issues by securely cloning the repository into an ephemeral Docker sandbox, generating and testing code patches, and seamlessly pushing the resulting fixes back to GitLab as fully documented Merge Requests.

## 2. Product Constitution

    Tech Stack & Platforms: Python (CLI framework), Docker Engine (local sandboxing), GitLab REST/GraphQL API, Git.

    Performance & Scalability: Lightweight CLI startup (< 2 seconds). The primary latency constraint will be LLM inference time and Docker container provisioning, which should be optimized for rapid teardown/spindown.

    Security & Compliance: Strict isolation of AI execution via Docker containers (no host OS manipulation). Authentication strictly via injected environment variables (e.g., GITLAB_TOKEN). No local caching of credentials.

    Architectural Constraints: Local execution environment requiring an active Docker daemon. Ephemeral workspace design—all cloned repositories and agent artifacts are purged locally upon execution completion or failure.

    Design Principles: Secure-by-default, Git-native workflow, minimal architectural bloat, and highly observable CLI output for developer monitoring.

## 3. Core Domain Entities

    GitLab Issue: Represents the target problem, encompassing the Issue ID, title, description, repository context, and associated metadata needed for the LLM to understand the goal.

    Sandbox Workspace: An ephemeral, isolated Docker container representing the execution environment. It contains the cloned Git repository, the language runtime, and serves as the boundary for all agent-executed bash commands and tests.

    Merge Request (MR): The final Git/GitLab artifact generated by the agent, containing the source branch, target branch, code diff, and an AI-generated summary of the problem, the root cause, and the implemented fix.

## 4. Feature Specifications

### Module 1: CLI Initialization & Context Gathering

    Feature 1.1: GitLab Issue Ingestion

        User Story: As a developer, I want to pass a GitLab issue URL or ID to the CLI so that the agent knows what problem to solve.

        Acceptance Criteria:

            CLI accepts an issue identifier as a primary argument.

            System validates the GITLAB_TOKEN environment variable.

            System fetches the issue details and repository metadata via the GitLab API.

        Business Logic/Rules: CLI must gracefully exit with a descriptive error if the token is missing/invalid, or if the user lacks read permissions for the target repository.

### Module 2: Secure Agent Execution

    Feature 2.1: Ephemeral Workspace Setup

        User Story: As a system architect, I want the agent to operate in an isolated environment so that my local machine is safe from destructive commands.

        Acceptance Criteria:

            CLI initializes a lightweight Docker container.

            Agent clones the target GitLab repository into the container using the PAT.

        Business Logic/Rules: Workspace must be completely torn down upon success, failure, or user interrupt (Ctrl+C).

    Feature 2.2: Autonomous Patch Generation Loop

        User Story: As a developer, I want the AI to iteratively navigate the codebase, write code, and run tests until the issue is fixed.

        Acceptance Criteria:

            Agent receives the issue context and executes shell commands strictly within the Docker sandbox.

            Agent edits files and validates syntax/tests within the sandbox.

        Business Logic/Rules: Maximum iteration limit must be enforced to prevent infinite loops and excessive LLM API costs.

### Module 3: Git & GitLab Output

    Feature 3.1: Automated Branching & Committing

        User Story: As a developer, I want the agent's final working state to be properly version-controlled.

        Acceptance Criteria:

            Upon successful patch generation, the agent creates a new branch (e.g., swe-agent/issue-[ID]).

            Agent commits the changed files with a descriptive commit message.

            Agent pushes the branch to the remote GitLab repository.

        Business Logic/Rules: Standard Git operations executed securely using the scoped GitLab PAT.

    Feature 3.2: Merge Request Generation

        User Story: As a code reviewer, I want a comprehensive summary of what the AI changed and why, so I can review the MR efficiently.

        Acceptance Criteria:

            CLI triggers a call to the GitLab API to open an MR from the new branch to the default branch.

            The MR description contains an AI-generated summary of the fix, referencing the original Issue ID.

        Business Logic/Rules: MR should be marked as Draft/WIP depending on configuration, and CLI should output the direct URL to the MR upon completion.

## 5. Out of Scope

    Support for GitHub, Bitbucket, or other version control systems.

    Execution directly inside GitLab CI/CD pipelines (deferred to a future phase).

    Multi-issue bulk processing (one execution = one issue).

    Long-running cloud server hosting of the agent.
